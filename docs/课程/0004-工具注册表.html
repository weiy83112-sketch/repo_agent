Lesson 0004

工具注册表（Tool Registry）

上一课讲了“模型决定调用工具，程序真正执行工具”。这一课讲下一层问题： 模型怎么知道系统里有哪些工具、每个工具该怎么用？

一句话： 工具注册表就是一张“工具说明书”，告诉模型有哪些工具、每个工具做什么、需要哪些参数。

1. 为什么需要工具注册表

模型不是自动知道你的项目里有 read_file、list_files 这些函数。 如果你不告诉它，它就不知道可以调用什么。

所以我们要把工具整理成一张清单，交给模型：

可用工具:
- list_files(path): 列出目录下的文件
- read_file(path): 读取文本文件内容
- search_text(query): 搜索项目中的关键词
- run_command(command): 执行允许的命令

2. 一个工具需要三类信息

信息 英文 作用
工具名 Name 模型用这个名字选择工具，例如 read_file。
工具描述 Description 告诉模型什么时候应该用这个工具。
参数结构 Schema 告诉模型要传哪些参数、参数是什么类型。

3. 以 read_file 为例

真实函数可能长这样：

def read_file(path: str) -> str:
    """读取一个文本文件，并返回文件内容。"""
    ...

给模型看的工具说明可以理解成：

{
  "name": "read_file",
  "description": "读取一个文本文件，并返回文件内容。",
  "parameters": {
    "path": {
      "type": "string",
      "description": "要读取的文件路径，例如 README.md"
    }
  }
}

模型看到用户问“这个项目怎么运行？”，就可能判断：我需要看 README， 所以选择 read_file，参数是 {"path": "README.md"}。

4. 工具注册表在 Agent 流程里的位置

用户问题
  -> 模型读取工具注册表
  -> 模型选择工具名和参数
  -> 程序执行工具
  -> 工具结果写回 State
  -> 模型继续分析

注意：工具注册表不是用来执行工具的。它只是告诉模型“有哪些工具可以选”。 真正执行工具的仍然是程序代码。

5. 为什么描述很重要

工具描述写得差，模型就容易选错工具。

差的描述 好的描述
read_file: read file read_file: 读取项目中的文本文件。适合查看 README、配置文件、源码片段。
search: search search_text: 在项目文件中搜索关键词，并返回匹配文件和行号。

6. 工具注册表和结构化输出的关系

工具调用要求模型不要随便写一段话，而要输出稳定结构： 

{
  "tool_name": "read_file",
  "arguments": {
    "path": "README.md"
  }
}

这就是后面要学的结构化输出（Structured Output）的一种实际用途： 让模型输出程序能读懂的格式。

7. 面试表达

我会把工具能力注册成带名称、描述和参数 schema 的工具表。模型只负责根据用户问题选择工具和参数， 程序根据工具名找到真实函数并执行。这样既能让模型理解可用能力，又能通过参数 schema 和校验减少错误调用。

8. 检查自己是否理解

问题 1：工具注册表是给谁看的？

主要是给模型看的，让模型知道有哪些工具、每个工具适合什么场景、需要什么参数。

问题 2：工具注册表会真正执行工具吗？

不会。它只是说明书。真正执行工具的是程序代码。

问题 3：为什么参数 schema 重要？

因为它约束模型必须给出程序能理解的参数，减少乱传参数和格式错误。
