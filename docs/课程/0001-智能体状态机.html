Lesson 0001

Agent 状态机（Agent State Machine）

这一课只抓一个核心：Agent 不是一次性问答，而是按步骤推进的工作流程。 LangGraph 把这个流程表达成 State、Node、Edge。

一句话： Agent 状态机就是把 Agent 做事的过程拆成一个个步骤，并规定每一步之后该往哪里走。

1. 为什么需要状态机

普通 LLM 问答通常是：用户提问，模型直接回答。Agent 要做的是更复杂的工作： 判断任务、读取文件、调用工具、分析结果、必要时继续查询，最后基于证据回答。 

普通问答:
用户问题 -> 模型直接回答

Agent 状态机:
用户问题 -> 判断任务 -> 调工具 -> 看结果 -> 决定下一步 -> 最终回答

2. 三个核心词

状态（State）

当前任务携带的信息，例如用户问题、项目路径、读过的文件、工具结果和最终回答。

节点（Node）

一个具体处理步骤，例如判断问题类型、读取 README、搜索报错关键词。

边（Edge）

步骤之间的连接规则。可以是固定下一步，也可以根据结果选择不同路线。

3. State：Agent 的工作台

State 不是数据库，也不是全局变量。它是一次任务执行过程中不断更新的上下文。 对我们的开源项目阅读 Agent 来说，State 可以长这样： 

{
  "question": "这个项目怎么运行？",
  "repo_path": "C:/projects/demo",
  "task_type": "run_project",
  "read_files": ["README.md", "requirements.txt"],
  "tool_results": [],
  "final_answer": ""
}

4. Node：每个节点只做一件事

一个好节点应该小而清楚。比如 classify_question 只负责判断用户问题类型， inspect_repo 只负责查看项目结构，final_answer 只负责组织回答。 

def classify_question(state):
    question = state["question"]
    task_type = model_router.simple(question)
    return {"task_type": task_type}

注意这里调用的是 model_router.simple，不是写死某个具体模型。 这和上一段学习里的 Model Router 思路是一致的。

5. Edge：决定下一步去哪

Edge 有两种常见形式：

类型 含义 例子
普通边（Edge） 一个步骤完成后固定去下一步。 inspect_repo -> choose_tool
条件边（Conditional Edge） 根据状态里的结果选择下一步。 如果还缺信息，继续调用工具；如果信息够了，生成最终回答。

6. 我们项目的第一版流程

START
  -> classify_question
  -> inspect_repo
  -> choose_tool
  -> run_tool
  -> analyze_result
  -> should_continue?
       -> yes: choose_tool
       -> no: final_answer
  -> END

翻译成人话就是：

  1. 先判断用户想问什么。
  2. 查看项目结构。
  3. 选择要使用的工具。
  4. 执行工具，例如读文件或搜索文本。
  5. 分析工具结果。
  6. 判断是否还需要继续查。
  7. 信息足够后生成最终回答。

7. 面试表达

我用 LangGraph 把 Agent 设计成状态机。State 保存当前任务上下文， Node 表示独立处理步骤，Edge 控制步骤之间的流转。这样 Agent 不只是一次性回答， 而是能根据中间结果持续调用工具、更新状态，并在信息足够时生成最终答案。

8. 检查自己是否理解

问题 1：State、Node、Edge 分别解决什么问题？

State 保存上下文，Node 执行一个步骤，Edge 决定步骤之间如何流转。

问题 2：为什么 Agent 不应该直接一次性回答所有问题？

因为真实任务经常需要读取外部信息、调用工具、根据结果继续判断。状态机能把这个过程拆开并管理起来。

问题 3：Conditional Edge 和普通 Edge 的区别是什么？

普通 Edge 固定去下一步；Conditional Edge 会根据 State 里的结果选择不同下一步。
