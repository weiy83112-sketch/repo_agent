Lesson 0006

Agent 为什么需要 messages

这一课不先背 LangGraph 名词。我们从你的真实目标出发：把 GitHub 上的开源 Agent 项目放进 source/，然后让 Agent 帮你读源码、读 README、总结怎么运行。

一句话： messages 就是 Agent 的工作记录本，记录“用户问了什么、模型想做什么、工具查到了什么、模型最后怎么回答”。

1. 先回忆前几课

前面几课其实已经拼出了半个 Agent：

知识点 你可以先这样理解
State Agent 做任务时携带的上下文。
Tool Calling 模型判断自己需要调用工具。
Tool Registry 告诉模型有哪些工具可以用。
Structured Output 让模型按固定格式输出工具名和参数。

现在还差一个问题：这些过程发生以后，Agent 把它们记在哪里？答案就是 messages。

2. 真实学习场景

假设你以后把一个开源 Agent 项目放到：

source/openhands/

然后你问自己的 Agent：

帮我看看 source/openhands 这个项目怎么启动

Agent 不能凭空回答。它应该先想：我要看这个项目的 README。

read_file("source/openhands/README.md")

工具读完 README 后，Agent 再根据真实内容回答你。

3. Agent 内部发生的四步

第 1 步：保存用户问题
第 2 步：模型决定调用 read_file
第 3 步：程序执行 read_file，拿到 README 内容
第 4 步：模型根据 README 内容回答用户

messages 的作用，就是把这四步一条一条记下来。

4. 先用普通 Python 表示

先不要管 LangChain 类名。我们只用 Python 的 list 和 dict 来理解。

messages = []

messages.append({
    "role": "user",
    "content": "帮我看看 source/openhands 这个项目怎么启动"
})

messages.append({
    "role": "assistant",
    "tool_call": {
        "name": "read_file",
        "args": {
            "path": "source/openhands/README.md"
        }
    }
})

messages.append({
    "role": "tool",
    "content": "README 里说启动命令是 docker compose up"
})

messages.append({
    "role": "assistant",
    "content": "这个项目可以通过 docker compose up 启动。"
})

5. 一行一行看

messages = []：创建一个空列表。这个列表就是 Agent 的工作记录本。

messages.append({...})：往记录本里新增一条记录。

"role": "user"：说明这条记录来自用户。

"content": "帮我看看 source/openhands 这个项目怎么启动"：这是用户原始问题。

"role": "assistant"：说明这条记录来自模型。

"tool_call": {...}：模型这次不是直接回答，而是说“我要调用一个工具”。

"name": "read_file"：模型选择的工具名是 read_file。

"args": {...}：模型给工具准备的参数。

"path": "source/openhands/README.md"：模型希望程序读取这个 README 文件。

"role": "tool"：说明这条记录来自工具执行结果。

"content": "README 里说启动命令是 docker compose up"：这是工具读文件后返回的真实信息。

最后一条 "role": "assistant"：模型看到了工具结果，所以现在可以回答用户。

6. messages 到底是什么

现在可以给 messages 下定义了：

messages = [
  用户问题,
  模型动作,
  工具结果,
  模型回答
]

所以它不是一个难词。它就是 Agent 运行过程的记录。

普通聊天只需要记“用户说了什么、模型回了什么”。Agent 还要记“模型调用了什么工具、工具返回了什么结果”。

7. 再对应到 LangChain 名词

理解了普通版本，再看 LangChain 术语就不吓人了。

普通理解 LangChain 名词 含义
role: user HumanMessage 用户发来的消息。
role: assistant，带 tool_call AIMessage 模型发出的消息，可以包含工具调用请求。
role: tool ToolMessage 工具执行后的结果消息。

8. 放回 Agent State

在 Agent 里，messages 通常会放进 state：

state = {
    "messages": messages
}

state 是整个任务状态。

messages 是状态里最重要的一部分。

下一个节点看到 state["messages"]，就知道前面发生了什么。

9. 和你的项目路线的关系

source/
  -> 放 GitHub 开源 Agent 项目

docs/
  -> 记录你从开源项目里学到的知识

project/
  -> 写你自己的 Agent 项目

所以后面我们不应该只抽象学概念，而应该围绕 source/ 里的真实开源项目来学。

10. 面试表达

我理解 Agent 的 messages 不是普通聊天记录，而是完整的执行轨迹。它会记录用户问题、模型发出的工具调用、工具返回的观察结果，以及模型基于观察结果生成的最终回答。这样后续节点和模型都能根据 State 里的 messages 继续推理。

11. 检查自己是否理解

问题 1：messages 为什么不只是聊天记录？

因为 Agent 还需要记录工具调用和工具结果，不只是用户和模型的普通对话。

问题 2：模型为什么不能直接回答 source/openhands 怎么启动？

因为它需要先读取真实 README，工具结果比凭空猜测更可靠。

问题 3：ToolMessage 对应普通版本里的哪一类记录？

对应 role: tool 的记录，也就是工具执行后的真实结果。
