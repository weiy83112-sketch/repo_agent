Lesson 0003

工具调用（Tool Calling）

这一课只讲一个核心：Agent 为什么不只是聊天机器人。因为它可以选择工具、调用工具、 读取工具结果，再决定下一步。

一句话： 工具调用就是让模型在需要外部能力时，调用我们提前定义好的函数或接口。

1. 为什么 Agent 需要工具

模型本身只会根据已有上下文生成文字。它不知道你电脑上的文件，也不能自己真的跑测试。 如果我们想让 Agent 读取项目、搜索代码、运行命令，就必须把这些能力做成工具。 

普通 LLM:
用户问 -> 模型猜答案

Agent:
用户问 -> 判断需要什么信息 -> 调工具 -> 观察结果 -> 再回答

2. 工具到底是什么

最简单地说，工具就是一个有名字、有参数、有返回值的函数。

def read_file(path: str) -> str:
    return 文件内容

如果 Agent 想知道 README.md 里写了什么，它可以调用：

read_file("README.md")

工具执行后，会返回文件内容。模型再根据这个真实内容继续分析。

3. 工具调用的四个步骤

  1. 我们先定义工具，例如 list_files、read_file、search_text。
  2. 模型看到用户问题后，判断需不需要调用工具。
  3. 如果需要，模型输出工具名和参数。
  4. 程序执行工具，把结果放回 State，模型再继续思考。

4. 以开源项目阅读 Agent 为例

用户问：

这个项目怎么运行？

Agent 不应该直接猜。它应该这样做：

1. 调用 list_files(".")
2. 发现 README.md 和 requirements.txt
3. 调用 read_file("README.md")
4. 调用 read_file("requirements.txt")
5. 根据真实文件内容回答怎么运行

5. 我们项目第一批工具

工具 作用 返回什么
list_files(path) 列出目录下有哪些文件。 文件名列表。
read_file(path) 读取一个文本文件。 文件内容。
search_text(query) 在项目里搜索关键词。 匹配到的文件和行号。
run_command(command) 运行受限制的命令，例如测试命令。 退出码、标准输出、错误输出。

6. 工具调用和状态机的关系

工具调用通常不是单独存在的，它会嵌进 Agent 状态机里。

choose_tool
  -> run_tool
  -> analyze_result
  -> should_continue?
       -> yes: choose_tool
       -> no: final_answer

也就是说，工具执行结果会更新 State。条件边再根据 State 判断：信息够了就回答， 不够就继续调用别的工具。

7. 一个重要边界

模型不能直接碰你的电脑。真正执行文件读取、搜索、命令运行的是程序代码。 模型只是提出“我要调用哪个工具，以及传什么参数”。

所以 Tool Calling 分成两层：

层 负责什么
模型 决定要不要调用工具、调用哪个工具、参数是什么。
程序 真正执行工具，并把结果交回模型。

8. 面试表达

我把外部能力封装成工具，例如读取文件、搜索代码、运行测试。Agent 在状态机中判断是否需要工具， 程序执行工具并把结果写回 State，模型再根据工具结果继续分析。这样 Agent 的回答不是凭空生成， 而是基于真实项目信息。

9. 检查自己是否理解

问题 1：工具和普通函数有什么关系？

工具本质上就是提供给 Agent 调用的函数，只是它有明确的名字、参数说明和返回结果。

问题 2：模型会直接读取你电脑上的文件吗？

不会。模型只决定要调用哪个工具，真正读取文件的是程序代码。

问题 3：工具结果应该放到哪里？

放回 State，给后面的节点和模型继续使用。
