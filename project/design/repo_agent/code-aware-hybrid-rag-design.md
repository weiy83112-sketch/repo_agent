# Code-Aware Hybrid RAG Design

## 1. 目标

将当前 CLI Repo Agent 深化为面向中大型 Python 仓库的代码问答 Agent，同时满足两个目标：

1. 提升对程序入口、功能位置、相关文件、类与函数调用关系等问题的实际回答效果。
2. 形成可复现、可评测、适合校招作品集和面试讲解的完整工程案例。

第一阶段目标仓库规模为约 1,000～10,000 个 Python 文件。Agent 必须优先依据真实代码回答，并在结果中提供文件路径、符号名称和代码行号。

## 2. 范围

### 2.1 第一阶段包含

- Python AST 代码切块。
- 本地持久化索引和增量更新。
- SQLite FTS5 关键词检索。
- 本地 Embedding 语义检索。
- 关键词与向量结果的混合排序。
- 上下文去重、邻近代码补充和预算控制。
- LangGraph 检索节点与现有文件工具协作。
- 两个 Python 仓库、约 30 道问题的改造前后评测。
- 回答中的代码证据引用。

### 2.2 第一阶段不包含

- Java、JavaScript 等多语言解析。
- Qdrant、Elasticsearch 等外部检索服务。
- 多 Agent 协作。
- Web 界面。
- 跨进程长期对话记忆。
- 完整静态调用图和知识图谱。

多轮摘要和调用关系图属于后续独立深化阶段，不阻塞第一阶段交付。

## 3. 选择的方案

采用“代码感知混合 RAG”，不采用纯向量 RAG，也不在第一阶段建立完整代码知识图谱。

- 关键词检索擅长函数名、类名、文件路径和精确标识符。
- 向量检索用于将中文自然语言问题匹配到英文代码和语义相近实现。
- AST 切块保证函数和类不会被固定长度切块随意截断。
- Context Builder 负责控制真正传给模型的内容，避免“检索到了”却因为上下文混乱而回答失败。

完整调用图具有价值，但解析导入别名、动态调用、装饰器和框架注册的成本较高，因此只保留扩展接口，不进入本阶段实现。

## 4. 总体架构

```text
目标 Python 仓库
    ↓
Repository Scanner
    ↓
Python AST Chunker
    ↓
Repository Index
    ├── SQLite 文件与代码块元数据
    ├── SQLite FTS5 关键词索引
    └── 本地 Embedding 向量

用户问题
    ↓
Hybrid Retriever
    ├── 关键词召回
    ├── 向量召回
    └── 混合排序
    ↓
Context Builder
    ├── 去重
    ├── 邻近代码补充
    ├── 文件多样性控制
    └── 上下文预算
    ↓
LangGraph call_model
    ↓
现有只读工具继续调查
    ↓
带路径、符号和行号证据的回答
```

## 5. 模块边界

### 5.1 Repository Scanner

职责：发现允许索引的 Python 文件，并判断文件是否发生变化。

输入：仓库根路径。

输出：文件路径、大小、修改时间和内容哈希。

规则：

- 只处理仓库边界内的 `.py` 文件。
- 跳过 `.git`、虚拟环境、缓存目录和常见构建产物。
- 复用当前 1 MiB 单文件限制。
- 使用内容哈希判断新增、修改和删除，修改时间只用于快速预筛选。

### 5.2 Python AST Chunker

职责：把一个 Python 文件转换成可检索的结构化代码块。

每个代码块至少包含：

- `chunk_id`
- 相对文件路径
- 符号完整名称
- 类型：module、class、function 或 method
- 起止行号
- 原始代码
- docstring
- 所属父符号
- 内容哈希

模块级代码、导入语句和无法归入函数或类的语句组成 module chunk。类和函数以 AST 的 `lineno`、`end_lineno` 为边界。过长符号可以在保持行号连续的前提下切成子块，但不得把不同符号混成一个块。

### 5.3 Repository Index

职责：统一负责索引写入、删除、更新和查询，不让上层依赖具体存储细节。

第一阶段采用单个 SQLite 数据库：

- 普通表保存文件、代码块和索引版本。
- FTS5 虚拟表保存可全文搜索的路径、符号、docstring 和代码文本。
- 归一化后的 `float32` Embedding 以 BLOB 保存；查询进程将当前向量集合加载为 NumPy 矩阵进行余弦相似度计算。

索引默认保存在 `Path.home() / ".repo_agent" / "indexes" / <repository_hash>`，其中 `repository_hash` 由规范化后的仓库绝对路径生成。不同仓库互相隔离，并且不会向被分析的仓库写入文件。存储层通过接口隔离，后续可以替换为独立向量数据库。

### 5.4 Local Embedder

职责：把问题和代码块转换为向量。

第一阶段通过 `sentence-transformers` 使用 `intfloat/multilingual-e5-small`，以支持中文问题和英文代码之间的匹配。模型名称仍由配置提供，方便后续替换并重新评测。模型首次使用时需要下载，下载完成后的索引和查询不依赖付费 API Key。

代码只依赖 `Embedder` 接口，不直接依赖具体模型类，便于后续替换代码专用模型并进行效果对比。

### 5.5 Hybrid Retriever

职责：根据用户问题返回统一排序后的候选代码块。

执行过程：

1. FTS5 默认召回前 20 个精确名称、路径和关键词候选。
2. 本地向量默认召回前 20 个语义候选。
3. 使用 `k=60` 的 Reciprocal Rank Fusion 合并两个排名。
4. 对同一符号的重复子块去重。
5. 限制单个文件最多保留 3 个候选块，最终默认返回前 10 个候选。

Retriever 返回结构化 `RetrievedChunk`，其中包含内容、分数、召回来源和证据信息；它不负责拼接模型 Prompt。

### 5.6 Context Builder

职责：从候选代码块中构造有限、可追溯的模型上下文。

规则：

- 按相关性加入代码块，达到配置预算后停止。
- 在预算允许时补充同一符号或相邻行上下文。
- 同一代码不得重复加入。
- 不在代码块中间任意截断；预算不足时舍弃整个低优先级块。
- 每段上下文包含统一证据头：相对路径、符号名称、起止行号和召回来源。

第一阶段默认使用 12,000 个估算 Token 的上下文预算，因为 DeepSeek 的精确 tokenizer 不属于当前项目依赖。预算可以通过配置修改，估算器通过接口隔离，并在 README 中明确这是近似值。

### 5.7 Tool Runtime

当前工具执行器主要传递 `repo_path`。为让新检索工具复用已打开的索引，引入 `ToolRuntime`：

```text
ToolRuntime
├── repo_path
└── retriever
```

现有 `list_files`、`read_file` 和 `search_text` 继续使用仓库路径；新增 `search_code` 使用 Retriever。工具注册表仍然负责 schema、参数校验和函数映射，模型不能指定索引文件位置。

### 5.8 LangGraph State 与节点

`AgentState` 在现有 `messages` 之外增加：

- `question`：本次用户原始问题。
- `retrieved_context`：Context Builder 生成的结构化上下文。

图流程调整为：

```text
START
  ↓
retrieve_context
  ↓
call_model
  ↓
route_after_model
  ├── tool_calls → execute_tools → call_model
  └── final answer → END
```

`retrieve_context` 只在每个问题开始时执行一次。`call_model` 将检索上下文放入本次模型输入，但不把整份上下文反复追加到 `messages`，避免每次工具循环都膨胀 State。模型如果需要继续定位代码，可以调用 `search_code` 或现有文件工具。

## 6. 索引生命周期

CLI 启动并确定仓库路径后：

1. 打开该仓库对应的本地索引。
2. 比较扫描结果与文件表。
3. 只重新解析和向量化新增或修改的文件。
4. 删除已经不存在的文件所对应的代码块。
5. 完成更新后再进入问答循环。

一次索引更新应作为一个事务提交。若更新失败，保留上一次完整可用索引，不能留下元数据、FTS 和向量彼此不一致的半成品。

## 7. 问答数据流

以问题“CLI 是怎样启动 LangGraph Agent 的？”为例：

1. Runner 把原始问题写入初始 State。
2. `retrieve_context` 调用 Hybrid Retriever。
3. 关键词检索命中 `cli`、`run_graph_agent`，向量检索补充语义相关入口代码。
4. Context Builder 去重并选择带路径和行号的代码块。
5. `call_model` 收到用户消息和检索上下文。
6. 模型可以直接回答，也可以继续调用 `read_file` 或 `search_code`。
7. 工具成功结果和错误仍作为 `role="tool"` 消息进入 State。
8. 最终回答引用实际文件、符号和行号。

## 8. 错误与降级策略

- 文件超限、编码错误或 AST 解析失败：跳过该文件，记录可见警告，继续索引其他文件。
- 单个文件向量化失败：该文件仍进入关键词索引，并记录语义索引缺失。
- 本地模型尚未下载或无法加载：索引命令给出明确错误和处理提示；已有关键词索引仍可用于只读降级查询。
- SQLite 事务失败：回滚本次更新，继续保留旧索引。
- 索引版本不兼容：要求重建索引，不静默读取错误结构。
- 检索无结果：模型收到明确的“未找到候选代码”，再使用现有文件工具调查，不伪造证据。
- 上下文超出预算：只在代码块边界舍弃低优先级内容，并记录实际使用预算。
- 工具参数和文件边界错误：继续沿用当前 `execute_tools` 的 Tool Message 错误路径。

## 9. 评测设计

选择两个中大型开源 Python 仓库，每个仓库约 15 道题，共约 30 道固定问题。题目覆盖：

- 程序或命令入口。
- 功能实现位置。
- 类与函数职责。
- 一段主要调用链。
- 与某功能相关的多个文件。

每道题保存：问题、预期文件、预期符号、人工参考答案和题目类别。

### 9.1 检索指标

- `Recall@5`：正确文件或符号是否出现在前 5 个候选中。
- `MRR`：第一个正确候选出现得是否足够靠前。
- 关键词、向量和混合检索的消融对比。

### 9.2 回答指标

- 人工正确性评分：0（错误）、1（部分正确）、2（正确）。
- 证据覆盖率：答案中的关键结论是否有对应路径、符号和行号。
- 无依据结论数量。
- 模型调用次数、工具调用次数、响应时间和估算上下文用量。

### 9.3 第一阶段验收标准

- 混合检索 `Recall@5` 达到 80% 以上，并比当前基线提高至少 20 个百分点。
- 最终回答平均正确性不低于 1.6/2，并高于改造前基线。
- 至少 90% 的最终回答提供可核查的代码证据。
- 单文件修改后只重建受影响文件的索引。
- Agent 和索引器均不修改被分析仓库。
- 建立索引后的单次检索在开发机上的 P95 小于 1 秒；首次索引耗时作为实测结果公开记录，不设置脱离硬件的固定门槛。

真实付费模型调用仍由用户决定何时执行。检索指标、索引测试和本地集成测试不依赖 DeepSeek 余额。

## 10. 测试策略

- 单元测试：扫描规则、AST 切块、行号、增量变更、FTS 查询、排名融合、去重和上下文预算。
- 集成测试：在临时 Python 仓库中完成首次索引、修改、删除、降级检索和证据输出。
- 回归评测：固定 30 道问题，对比当前 Agent、关键词检索、向量检索和混合检索。
- 安全测试：路径逃逸、超大文件、非法 UTF-8、语法错误和索引损坏。

测试不直接调用付费模型；模型回答评测单独运行并保存时间、模型名称和配置，避免与确定性的检索测试混在一起。

## 11. 教学与实施顺序

按照项目现有 manual-first 决策，核心模块由教学带领完成，小型参数检查和重复胶水代码可以由 Agent 补齐并明确标注。

实施顺序：

1. 建立当前文件工具的评测基线和题目格式。
2. 定义 `CodeChunk`、`RetrievedChunk` 等数据结构。
3. 实现 Repository Scanner 和 AST Chunker。
4. 实现 SQLite Repository Index 和增量更新。
5. 实现关键词检索。
6. 接入本地 Embedder 和向量检索。
7. 实现混合排序与 Context Builder。
8. 引入 Tool Runtime 和 `search_code`。
9. 增加 LangGraph `retrieve_context` 节点。
10. 运行回归评测并更新 README、架构图和作品集结果。

第一课从“评测基线为何必须先于 RAG 实现”开始。这样每增加一种能力，都能证明它解决了什么问题，而不是只证明项目安装了一个新依赖。
